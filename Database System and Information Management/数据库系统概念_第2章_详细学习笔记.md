## 2.1 关系数据库的结构

### 2.1.1 什么是关系模型？

关系模型是数据库系统中最广泛使用的一种数据模型，由IBM的Edgar Codd于1970年提出。它之所以流行，是因为**概念简单**——用**表（table）**来组织数据。

#### 核心概念对照表

| 关系模型术语 | 日常用语 | 类比Excel | 说明 |
|-------------|---------|----------|------|
| **关系（Relation）** | 表 | 工作表 | 数据的集合，用二维表表示 |
| **元组（Tuple）** | 行/记录 | 一行数据 | 表中的一行，代表一个实体 |
| **属性（Attribute）** | 列/字段 | 一列 | 表中的一列，代表一种特征 |
| **域（Domain）** | 取值范围 | 数据类型 | 属性允许的取值集合 |

### 2.1.2 具体示例

#### 示例1：instructor（教师）表

| ID | name | dept_name | salary |
|----|------|-----------|--------|
| 10101 | Srinivasan | Comp. Sci. | 65000 |
| 12121 | Wu | Finance | 90000 |
| 15151 | Mozart | Music | 40000 |
| 22222 | Einstein | Physics | 95000 |

**分析：**
- 这是一个**关系**（relation），名为instructor
- 表中有4行，即有**4个元组**
- 每行代表一位教师的信息
- 表有4列，即有**4个属性**：ID、name、dept_name、salary
- ID属性的**域**是所有可能的教师编号
- salary属性的**域**是所有可能的工资数值

#### 示例2：course（课程）表

| course_id | title | dept_name | credits |
|-----------|-------|-----------|---------|
| BIO-101 | Intro. to Biology | Biology | 4 |
| CS-101 | Intro. to Computer Science | Comp. Sci. | 4 |
| CS-190 | Game Design | Comp. Sci. | 4 |
| FIN-201 | Investment Banking | Finance | 3 |

#### 示例3：prereq（先修课程）表

| course_id | prereq_id |
|-----------|-----------|
| BIO-301 | BIO-101 |
| CS-190 | CS-101 |
| CS-315 | CS-101 |
| CS-347 | CS-101 |

**这个表表示课程之间的先修关系：**
- BIO-301（遗传学）的先修课是BIO-101（生物学导论）
- CS-190（游戏设计）的先修课是CS-101（计算机科学导论）
- 以此类推...

### 2.1.3 关系的重要特性

#### 特性1：元组的无序性

**关键理解：** 关系是**元组的集合（set）**，集合中的元素没有顺序。

```
以下两种表示是同一个关系：

表示A（按ID排序）：          表示B（无序）：
ID: 10101, 12121, 15151     ID: 22222, 12121, 32343
     ↓                         ↓
ID: 22222, 32343, ...       ID: 10101, 15151, ...

✓ 它们是同一个关系，因为包含相同的元组集合
```

**实际意义：** 数据库不保证返回数据的顺序，除非你用`ORDER BY`明确指定。

#### 特性2：属性的原子性（Atomic）

**定义：** 如果一个域中的元素被认为是**不可再分的单元**，则该域是原子的。

**正例（原子）：**
```
name = "张三"          ✓ 作为一个整体使用，不可再分
salary = 65000         ✓ 一个数值，原子
phone = "13800138000"  ✓ 如果作为单个字符串存储，是原子的
```

**反例（非原子）：**
```
phone_numbers = "13800138000, 13900139000, 13700137000"
                     ✗ 包含多个电话号码，可以拆分

address = "北京市海淀区中关村大街1号"
                     ✗ 如果拆分成省、市、区、街道，则是非原子的
```

**注意：** 是否原子取决于**使用方式**：
- 如果你总是把电话号码当作一个整体来用 → 原子的
- 如果你需要分别提取国家码、区号、本地号码 → 非原子的

#### 特性3：空值（Null）

**定义：** 空值是一个特殊的值，表示**值未知**或**不存在**。

**示例：**
```
假设instructor表增加phone_number属性：

| ID    | name    | dept_name  | salary | phone_number |
|-------|---------|------------|--------|--------------|
| 10101 | Srinivasan | Comp. Sci. | 65000  | 555-1234     |
| 12121 | Wu      | Finance    | 90000  | NULL         |  ← 电话未知
| 15151 | Mozart  | Music      | 40000  | NULL         |  ← 没有电话
```

**警告：** 空值会带来很多问题！
- `NULL = NULL` 的结果是**未知**，不是**真**
- `NULL + 5` 的结果是**NULL**
- 聚合函数（如SUM、AVG）会忽略NULL值

**建议：** 尽量避免使用空值，可以用默认值或单独的标志字段。

## 2.2 数据库模式（Schema）vs 实例（Instance）

### 2.2.1 类比理解

| 概念 | 类比C语言 | 类比Java | 说明 |
|------|----------|---------|------|
| **关系模式** | `struct Student {int id; char name[20];}` | `class Student {int id; String name;}` | 定义结构（类型） |
| **关系实例** | `Student s = {101, "Alice"};` | `Student s = new Student(101, "Alice");` | 具体的数据（值） |

### 2.2.2 正式定义

#### 关系模式（Relation Schema）
- 描述关系的**逻辑结构**
- 包含：**属性列表** + **各属性的域**
- 类比：变量的**类型定义**
- 特点：**不常变化**

#### 关系实例（Relation Instance）
- 关系的**特定数据快照**
- 包含：**一组特定的元组**
- 类比：变量的**当前值**
- 特点：**随时间变化**

### 2.2.3 示例对比

```
【关系模式】instructor(ID, name, dept_name, salary)
           ↓
    定义了表的结构：
    - 有4个属性
    - ID是字符串类型
    - salary是数值类型
    - ...

【关系实例】2024年3月17日的instructor表
           ↓
    具体的数据：
    - 第1行：(10101, Srinivasan, Comp. Sci., 65000)
    - 第2行：(12121, Wu, Finance, 90000)
    - ...共12个元组
```

### 2.2.4 数据库模式与实例

| 概念 | 定义 | 类比 |
|------|------|------|
| **数据库模式** | 数据库的逻辑设计 | 建筑的蓝图 |
| **数据库实例** | 给定时刻数据库中的数据快照 | 建筑的实际状态 |

### 2.2.5 大学数据库完整模式

```sql
classroom(building, room_number, capacity)
    -- 教室信息：楼名、房间号、容量

department(dept_name, building, budget)
    -- 系信息：系名、所在楼、预算

course(course_id, title, dept_name, credits)
    -- 课程信息：课程号、标题、开课系、学分

instructor(ID, name, dept_name, salary)
    -- 教师信息：ID、姓名、所属系、工资

section(course_id, sec_id, semester, year, building, room_number, time_slot_id)
    -- 课程段信息：课程号、段号、学期、年份、教室、时间片

teaches(ID, course_id, sec_id, semester, year)
    -- 授课关系：教师ID、课程号、段号、学期、年份

student(ID, name, dept_name, tot_cred)
    -- 学生信息：ID、姓名、所属系、总学分

takes(ID, course_id, sec_id, semester, year, grade)
    -- 选课关系：学生ID、课程号、段号、学期、年份、成绩

advisor(s_ID, i_ID)
    -- 导师关系：学生ID、教师ID

time_slot(time_slot_id, day, start_time, end_time)
    -- 时间片信息：时间片ID、星期几、开始时间、结束时间

prereq(course_id, prereq_id)
    -- 先修关系：课程号、先修课程号
```

**注意属性名的重复：**
- `dept_name` 出现在多个关系中（instructor, course, department等）
- 这种重复是**有意的设计**，用于建立关系之间的联系

## 2.3 码（Keys）—— 本章重点！

### 2.3.1 为什么需要码？

**问题：** 如何区分关系中的不同元组？

**答案：** 使用**码**——能够唯一标识元组的属性集。

### 2.3.2 超码（Superkey）

#### 定义
**超码**是一个或多个属性的集合，将这些属性组合在一起可以**唯一地标识**出关系中的一个元组。

#### 示例分析

以instructor表为例：

| ID | name | dept_name | salary |
|----|------|-----------|--------|
| 10101 | Srinivasan | Comp. Sci. | 65000 |
| 12121 | Wu | Finance | 90000 |
| 15151 | Mozart | Music | 40000 |
| 22222 | Einstein | Physics | 95000 |

**判断哪些是超码：**

| 属性集 | 是否超码 | 原因 |
|--------|---------|------|
| {ID} | ✓ 是 | 每个教师ID唯一 |
| {name} | ✗ 否 | 可能有重名 |
| {ID, name} | ✓ 是 | ID已足够，加上name还是唯一 |
| {ID, salary} | ✓ 是 | ID已足够 |
| {name, dept_name} | ? 可能 | 如果同系没有重名，则是超码 |
| {dept_name} | ✗ 否 | 一个系有多名教师 |

**重要性质：**
- 如果K是超码，那么K的**任意超集**也是超码
- 例如：{ID}是超码 → {ID, name}、{ID, name, salary}都是超码

### 2.3.3 候选码（Candidate Key）

#### 定义
**候选码**是**最小的超码**——它的任意**真子集**都不是超码。

#### 理解"最小"

```
{ID, name} 是超码，但不是候选码
    ↓
因为 {ID}（真子集）已经是超码了
    ↓
所以 {ID, name} 不是最小的，不是候选码

{ID} 是超码，也是候选码
    ↓
因为它的真子集是空集，不是超码
    ↓
所以 {ID} 是最小的，是候选码
```

#### 示例

**instructor表的候选码：**
1. **{ID}** —— 候选码（假设ID唯一）
2. **{name, dept_name}** —— 可能是候选码（如果同系没有重名）

**注意：**
- 一个关系可以有**多个候选码**
- 候选码之间是**平行的**，没有主次之分

### 2.3.4 主码（Primary Key）

#### 定义
**主码**是被数据库设计者选中作为**主要方式**来区分不同元组的**候选码**。

#### 特点
- 一个关系只能有**一个主码**
- 主码的取值**不能重复**
- 主码的取值**不能为NULL**

#### 表示方法

**方式1：下划线**
```
instructor(<u>ID</u>, name, dept_name, salary)
         ↑
       主码属性加下划线
```

**方式2：放在最前面**
```
department(<u>dept_name</u>, building, budget)
    ↑
  主码属性通常列在最前面
```

#### 主码约束（Primary Key Constraint）

```
规则：关系中任意两个不同的元组，不能在主码属性上取值相同

违反示例：
| ID    | name    | dept_name  |
|-------|---------|------------|
| 10101 | Alice   | Comp. Sci. |
| 10101 | Bob     | Physics    |  ← 错误！ID重复了

正确示例：
| ID    | name    | dept_name  |
|-------|---------|------------|
| 10101 | Alice   | Comp. Sci. |
| 10102 | Bob     | Physics    |  ← OK，ID唯一
```

#### 如何选择主码？

**好的主码应该：**
1. **值不常变化** —— 如社会保障号、学号、员工号
2. **值唯一确定** —— 不会重复
3. **尽量简单** —— 单属性优于多属性组合

**不好的选择：**
- 姓名（可能重名）
- 地址（会变化）
- 电话号码（会变化）

**实际例子：**
- 中国学生：学号、身份证号
- 美国学生：Social Security Number (SSN)
- 跨国企业：企业自己生成的唯一标识符

### 2.3.5 多属性主码

有些情况下，单个属性不足以唯一标识元组，需要多个属性组合。

#### 示例1：classroom（教室）

```
classroom(<u>building</u>, <u>room_number</u>, capacity)
          ↑              ↑
        主码由两个属性组成
```

**为什么？**
- 单独`building`不够：一栋楼有多个房间
- 单独`room_number`不够：不同楼可能有相同房间号
- 组合起来唯一："Taylor楼101室"是唯一的

#### 示例2：time_slot（时间片）

```
time_slot(<u>time_slot_id</u>, <u>day</u>, <u>start_time</u>, end_time)
          ↑                ↑      ↑
        主码由三个属性组成
```

**为什么？**
- 一个时间片ID可能对应多天的多个时段
- 例如：'A'时间段 = 周一、周三、周五的8:00-8:50
- 需要`time_slot_id + day + start_time`才能唯一确定一个时段

### 2.3.6 外码（Foreign Key）

#### 现实问题

假设我们要查询：**"在Watson楼工作的所有教师"**

**分析：**
- instructor表有dept_name（系名）
- department表有building（楼名）和dept_name
- 需要通过dept_name关联两个表

**如何保证数据一致性？**
- 如果instructor中有个dept_name="Musicology"
- 但department表中没有这个系
- 这就出现了**数据不一致**！

#### 外码的定义

```
【正式定义】
设关系r1的属性集A引用关系r2的属性集B：
- r1称为引用关系（referencing relation）
- r2称为被引用关系（referenced relation）
- A称为外码（foreign key）

约束条件：
r1中每个元组在A上的取值，必须等于r2中某个元组在B上的取值
```

#### 外码示例

**示例1：instructor → department**

```
instructor(ID, name, <u>dept_name</u>, salary)
                     ↑
                   外码（引用department）

department(<u>dept_name</u>, building, budget)
           ↑
         被引用的主码
```

**约束：**
- instructor中的dept_name必须是department中存在的系
- 不能出现"Ghost Department"（幽灵系）

**示例2：section → classroom**

```
section(course_id, sec_id, semester, year, <u>building</u>, <u>room_number</u>, time_slot_id)
                                         ↑              ↑
                                       外码（共同引用classroom）

classroom(<u>building</u>, <u>room_number</u>, capacity)
          ↑              ↑
        被引用的主码（组合主码）
```

**约束：**
- section中的(building, room_number)组合必须是classroom中存在的教室

#### 外码 vs 被引用属性

**重要规则：**
- 被引用的属性必须是**主码**
- 外码可以引用**另一个关系**或**同一关系**（自引用）

**自引用示例：**
```
employee(<u>employee_id</u>, name, ..., manager_id)
                              ↑
                            外码，引用同一关系的employee_id
                            （表示员工的经理是谁）
```

### 2.3.7 引用完整性约束（Referential Integrity Constraint）

#### 外码的局限

外码要求被引用的属性必须是**主码**。

但有些情况下，我们想引用**非主码**的属性：

```
time_slot(<u>time_slot_id</u>, day, start_time, end_time)
          ↑
        主码是(time_slot_id, day, start_time)的组合

section(..., time_slot_id)
          ↑
        只想引用time_slot_id
        但time_slot_id本身不是主码！
```

#### 引用完整性约束

**定义：** 引用完整性约束是外码约束的**泛化版本**。

```
【引用完整性约束】
引用关系中的任意元组在指定属性上的取值，
必须出现在被引用关系中至少一个元组的指定属性上。

【与外码的区别】
- 外码：被引用属性必须是主码
- 引用完整性：被引用属性可以是任意属性
```

#### 实际应用

大多数数据库系统**只支持外码约束**，不支持一般的引用完整性约束。

## 2.4 模式图（Schema Diagram）

### 2.4.1 什么是模式图？

模式图是数据库模式的**图形化表示**，直观展示：
- 有哪些关系（表）
- 每个关系有哪些属性
- 主码是什么
- 外码约束是什么

### 2.4.2 模式图的表示规范

| 元素 | 表示方法 |
|------|---------|
| 关系 | 矩形框 |
| 关系名 | 框顶部，灰色背景 |
| 属性 | 框内列出 |
| 主码 | 属性加**下划线** |
| 外码约束 | **单箭头**从外码指向被引用主码 |
| 引用完整性约束 | **双箭头** |

### 2.4.3 示例模式图（文字版）

```
┌─────────────────────────────────────┐
│           instructor                │
├─────────────────────────────────────┤
│  <u>ID</u>                                │
│  name                               │
│  dept_name  ─────────┐              │
│  salary              │              │
└──────────────────────┼──────────────┘
                       │
                       ↓ (外码约束)
              ┌─────────────────────┐
              │     department      │
              ├─────────────────────┤
              │  <u>dept_name</u>         │
              │  building           │
              │  budget             │
              └─────────────────────┘
```

    │

## 2.6 关系代数

### 2.6.1 什么是关系代数？

关系代数是一组**运算**的集合：
- 输入：一个或两个关系
- 输出：一个新关系

**特点：**
- 运算对象是**关系**
- 运算结果也是**关系**
- 可以**嵌套**组合

### 2.6.2 运算分类

| 分类 | 运算 | 符号 | 输入数量 |
|------|------|------|---------|
| **一元运算** | 选择 | σ (sigma) | 1个关系 |
| | 投影 | π (pi) | 1个关系 |
| | 更名 | ρ (rho) | 1个关系 |
| **二元运算** | 并 | ∪ | 2个关系 |
| | 集差 | − | 2个关系 |
| | 笛卡儿积 | × | 2个关系 |
| | 连接 | ⋈ | 2个关系 |

### 2.6.3 选择运算（Selection）

#### 定义
选择运算选出满足给定**谓词（条件）**的元组。

#### 符号
- **σ**（小写希腊字母sigma）
- 谓词写在σ的下标
- 关系写在σ后的括号中

#### 语法
```
σ<谓词>(关系)
```

#### 示例

**示例1：选出物理系的教师**

```
σ_dept_name="Physics"_(instructor)

结果：
| ID    | name     | dept_name | salary |
|-------|----------|-----------|--------|
| 22222 | Einstein | Physics   | 95000  |
| 33456 | Gold     | Physics   | 87000  |
```

**示例2：选出工资大于80000的教师**

```
σ_salary>80000_(instructor)

结果：
| ID    | name     | dept_name  | salary |
|-------|----------|------------|--------|
| 12121 | Wu       | Finance    | 90000  |
| 22222 | Einstein | Physics    | 95000  |
| 33456 | Gold     | Physics    | 87000  |
| 83821 | Brandt   | Comp. Sci. | 92000  |
```

**示例3：复合条件（AND）**

```
σ_dept_name="Comp. Sci."_∧_salary>70000_(instructor)

结果：
| ID    | name   | dept_name  | salary |
|-------|--------|------------|--------|
| 83821 | Brandt | Comp. Sci. | 92000  |
```

#### 选择运算的特点

1. **一元运算**：只在一个关系上操作
2. **水平筛选**：选出满足条件的**行**
3. **保持所有列**：结果有与原关系相同的属性
4. **结果可能为空**：如果没有元组满足条件

## 📝 本章重点总结

### 必须掌握的概念

1. **关系模型的基本结构**
   - 关系 = 表
   - 元组 = 行
   - 属性 = 列

2. **模式 vs 实例**
   - 模式 = 结构定义（不常变）
   - 实例 = 具体数据（常变化）

3. **码的层次**
   ```
   超码 ⊃ 候选码 ⊃ 主码
   
   超码：能唯一标识（可能有冗余属性）
   候选码：最小超码（去掉任何属性就不行了）
   主码：被选中的候选码（实际使用）
   ```

4. **外码的作用**
   - 建立关系间的联系
   - 维护数据一致性
   - 防止"孤儿"数据

5. **关系代数基础**
   - 选择运算 σ：筛选行
   - 结果是新关系，可以嵌套

### 常见误区

| 误区 | 正确理解 |
|------|---------|
| "主码就是候选码" | 主码是**被选中的**候选码，一个关系可能有多个候选码，但只有一个主码 |
| "外码必须是单属性" | 外码可以是**属性组合**，如(building, room_number) |
| "关系是有序的" | 关系是**集合**，元组无序；只有明确ORDER BY才有序 |
| "NULL等于NULL" | NULL = NULL 结果是**未知**，不是真 |
